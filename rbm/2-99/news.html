DR2's future laid out

  Andrew Barry, author of RB, recently said a few things about DR2's future:

  • DR2's feature set is pretty much frozen, so there won't be any huge changes.
  • Databases will be improved over the course of the next few releases.
  • The Win32 compiler is, and will continue to get progressively stronger.
  • DR2 will become a respectable beta version sometime soon.

Meet ByRef and ByVal

  In DR2r37, we all welcome the ByVal and ByRef keywords to RB. These keywords go before a variable in a method's declaration (ie, Sub DoSomething(ByRef s as string)). They control how the variable is passed: By Reference, or By Value. You can get acquainted with them in the tips section.

Windows Improve

  At first, the Windows compiler really stunk. It crashed about 90% of the time, and only worked with extremely simple projects. However, there have been a lot of recent improvements to the compiler, and it just keeps on getting better. Some noticable improvements this month include a wide array of bug fixes, QuickTime compatibility (which is still very buggy), and support for a lot more controls.

  As a simple test, I recently compiled a program that generated tournament brackets and seedings for windows, and it worked flawlessly. Granted, it ran about 50 times slower than the mac counterpart, but this is probably more Virtual PC's fault than it is RB's.

  A notable difference is the ability to generate MDI applications. MDI (Multiple Document Interface) is a system that is employed by many windows programs that allows them to put many document windows into one big window. RB doesn't support any of MDI's more advanced features, but that will probably be added soon.

  However, as many people who have tried to use RB's MDI functionality have learned, it doesn't work too well. I compiled a simple program using MDI (you can turn it on in the Build Application dialog), and when I ran it in Windows, it hung up the system. DR2r59 claims to fix this, but I haven't had an opportunity to test it yet.

Ye Wee Improvements

  Since our last issue, there have been a lot of improvements to DR2 that were quite useful, but not worthy of a whole section in the news.

  One of these is block [de]commenting. Introduced in DR2r44, this allows you to comment or decomment an entire group of lines with a single keystroke. Simply select the lines, and press Cmd-'. RB adds a “'” mark to the start of each one. Press it again, and they're removed. Unfortunately, if a line is already commented, a “'” is not added, but it's removed when you decomment.

  Another feature worthy of note is the object destructor. This is virtually identical to a constructor (which I discussed last issue), but it fires when the object closes.

  A feature that's simply a convenience was added in r43. There is now an option under the Editor Settings dialog that lets you automatically hide the palettes when a code editor is visible. It's very useful, but it still has quirks. For example, if you open the New Variable dialog, they show up again. It also happens if you click on the code editor window from another application. Hopefully though, this will be fixed soon. I still vote for a "Show/Hide All Palettes" command. I've made a macro for it, and it's extremely useful.

New Tabpanel features

  The Tabpanel control is the only thing right now that you could possibly consider a container control in RB. And, with DR2r38, it just got better. You can now set it to use large or small tabs, set the font and font size, as well as the orientation (though you'll need OS 8.5). The lovely screenshot to this article's right shows its incredibly funky new abilities. It does have some problems though. On horizontally facing tabs, you can only fit about one or two characters in. Of course, it's still much better than the pre-r38 one, which had very, very few options.

RB goes color model crazy!

  Beginning in DR2r37, RB can handle color models OTHER than RGB! It can also display a color using the SelectColor <color> command. It has an odd syntax actually. You'd think that you would type c=SelectColor, but Andrew decided that he needed a way to test ByVal and ByRef (see another news article) and thus, the odd syntax. However, it returns a boolean, so you'll know if the user clicked cancel or not.

  Also, the color object now has Hue, Saturation, Value, Cyan, Magenta, and Yellow properties. They range from 0.0 to 1.0 (0=0%, 1=100%). And, like the Red, Green and Blue properties, they have methods as well: HSV(hue as double, saturation as double, value as double) As Color and CMY(C as double, M as double, Y as double) As Color. They work exactly like the RGB function works.

  At this point, you may be asking yourself, where's the K? Why only CMY? Simply enough, you can display any color on the screen using only C, M, and Y. K is only used to add quality to printouts.

Reference Window gets a makeover

  In r54, the reference window got a major new appearance that should be considered a giant step backwards. Now, instead of the standard five-popupmenu style that we've had since the mid-DR1s, we have a large list on the left (similar to the code editor) that has 25 folders in it. To get help on a topic, you must find the appropriate one (not an easy thing), and then search through the list of items that appear in small print under it.

  If you press the "Alpha" button, all of the items appear without the folders, but you can't use the keyboard to select items.

  Thankfully, some of the reference has been updated for DR2 though, but we still can't control the font used in the reference, and it still scrolls at an incredibly slow rate.
Properties gets a makeover

  In DR2r39, the properties window got a long-awaited makeover. The changes aren't particularly large, but it uses shades of gray instead of the dotted outline that was everywhere in the old properties window. Also, it has a few very practical uses: It's faster, and you can tab between properties.

  If you make a window with several controls, and tab between them, you'll notice that the properties window refreshes noticable faster, and flickers far less than the old one.

  Also, if you select a property, and press either tab or shift-tab,you'll move to another property, as opposed to another control, which is what happened in the pre-DR2r39 properties window.

  The new properties window is expected to be just the first in a large group of appearance changes, so keep you eyes open for a new code editor.
Toolbar gets a rearrangement

  As was promised by Andrew Barry, the RB interface is changing. The second such change was to the toolbar, which is now wider and shorter. Instead of having two tools per row, it now has three. This means that it has far more empty spaces than it did before. Personally, I don't think this is an improvement. It simply means more wasted space. And, horizontal space is a lot more important than vertical space in RB (after all, you have to cram a code editor in there somewhere). Anyway, for those who haven't downloaded DR2r42, there are screenshots surrounding this news article. The old is on the left. The new one is on the right. As you can see, the new version crams more controls into a similiar vertical space, but takes up more room horizontally, and wastes more space on blank squares.Which one is better? I tend to prefer the new one, but opinions are bound to differ.
QuickDrawLib hits 2.0

  Einhuger software has been widely known for their many plugins (most of which give you access to the Apple Toolbox), and other classes such as cTableControl.

  Now, as the plugin format has drastically improved in DR2, Bjorn has been busily updating his plugins to take advantage of these new features. QDLib 2.0, which has been available in beta form for registered users for the past two months, has finally been released, and it's great.

  Now, thanks to the new DR2 features, it uses standard Apple Toolbox objects as well as methods. Not only does this simplify things a great deal, it also makes it much more intuitive. Registered users can get this free upgrade from the Einhuger site.

The Month of the Bug

  A new web site has been launched: REALbugs. It can be found at realbugs.dynip.com, and is dedicated (as its name might imply) to tracking, reporting, and discussing bugs in RB. The collection isn't particularly large at this point, but it's growing, and the site's author appreciates any submissions.

  Unfortunately, it closed a few weeks later, but it caught REAL SW's notice. They are revamping their bug-fixing efforts, and will provide an application called "REALbugs" that allows you to view and search all reported bugs, so that you won't report a bug that's already been reported. This will make their lives MUCH easier.

SuperSocketLite goes beta

  James Milne's much-anticipated SuperSocket plugin has gone beta. The full version does not have an available beta yet, but it will go beta soon. You can get your copy of the beta version from Essence SW's site. SuperSocket offers several key improvements over RB's built-in sockets: It supports UDP and AppleTalk, as well as TCP/IP; it has built-in support for multiple connections, which make it great for servers; and it allows DNS resolution—something RB's sockets can't do (they'll resolve the DNS, they just won't let you see the IP). When it's released, SuperSocket will cost $100 ($40 academic).

REALnews reaches the world

  In general, staying on top of news in the RB world is very difficult to do on a daily basis. Sure, you can read RBM's news section (which I strongly encourage =) ), but I tend to only cover major events from the past month (or the past two months, as is the case with this issue).

  However, if you just want to get a quick check of the day's news, then REALnews, a wonderful new web site, is just for you. It provides daily updates that include product announcements, new versions of RB, requests, and loads of other things. I've got it bookmarked, and you should too.
